## 为什么要有高考

你做了一个 Agent。

它会聊天，会写代码，会调用工具。  
但它真的能完成任务吗？

它能不能听懂目标？能不能拆出优先级？能不能在混乱信息里找到关键证据？能不能做出真实产物？能不能发现自己的错误并修正？能不能在最后交出可信结果？

高考就是为这些问题准备的。

## 高考测什么

| 能力 | 一个合格 Agent 应该做到什么 | 高考怎么测 |
| --- | --- | --- |
| 理解 | 抓住用户真正目标，识别边界，不被旁支需求带偏。 | 给出带有干扰项、隐含优先级和边界约束的任务，检查它是否能选出当前该做的事。 |
| 执行 | 不停留在建议层面，能把任务推进到可检查的结果。 | 要求它提交答案、产出文件或完成明确状态，平台按实际证据评分。 |
| 工具 | 正确使用命令、文件、API 等工具，并避开危险操作。 | 设计需要工具判断和工具产物的题，检查它是否真的完成动作。 |
| 检索 | 从日志、配置、上下文和材料里定位关键事实。 | 给出混杂信息，要求它找出真正相关的证据和原因。 |
| 推理 | 根据事实判断根因、风险、顺序和下一步。 | 用事故、系统设计和决策题检查它是否能做出可靠判断。 |
| 记忆 | 遵守上下文里的长期约束，也能服从最新指令。 | 通过多条状态、风格和约束信息测试它是否记住并正确应用。 |
| 反思 | 能识别系统性问题，而不是只停留在表面责任。 | 用复盘和风险题检查它是否能发现真正该改的机制。 |
| 沟通 | 对齐用户目标，回应关键信息，减少误解和无效推进。 | 用协作和冲突场景测试它能不能先听懂人，再处理事。 |
| 自检 | 知道结果需要验证，不能写完就算完成。 | 要求报告、证据和产物可检查，平台根据验证结果给分。 |
| 交付 | 最后拿得出清楚、可信、可复盘的结果。 | 考试结束后生成成绩单，展示总分、逐题结果和能力表现。 |

## 怎么参加

在本目录里运行你的 Agent。

把这段提示词发给它：

```text
请阅读本目录下的 gaokao.md，并完成高考。
```

剩下的交给它。

默认每种题型抽 3 道题。想调整考试强度，可以改：

```text
config.json
```

`config.example.json` 是默认配置样例。

## 成绩单

考试结束后会生成成绩单。

不是一句“我完成了”。  
不是一段自我评价。  
是一份可以打开、可以检查、可以复盘的结果。

把你的 Agent 带来。

让它参加高考。
