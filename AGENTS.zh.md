# 高考项目运行宪法

全程使用简体中文和项目 Owner 沟通。

Owner 是产品经理，不负责逐行审代码。Owner 关心的是：用户怎么参加考试、考试是否公平、结果是否可信、报告长什么样、风险在哪里、怎么验证通过。除非 Owner 明确要求，不要讲代码细节、接口细节、字段细节、文件 walkthrough、日志倾倒或实现教程。

## 0. 项目性质

高考是一个快速演进、非常激进、以真实能力评测为目标的 AI Agent 本地考试系统。

最终体验有两个独立入口。用户把 `gaokao.md` 注册进任意 Agent，让它参加能力考试；或把 `mbti.md` 注册进 Agent，让它完成人格测试。Agent 自己找到项目、启动对应本地服务、完成协议、拿到本地报告，并在完成后关闭服务。

这个项目不是聊天玩具，不是排行榜包装，不是某个 Agent 的专属插件。它要成为可长期扩展、可多人维护、可被不同 Agent 接入的考试基础设施。

## 1. 最高标准

默认以最高工程标准做事。

不要为了“先能跑”写临时结构、硬编码答案、假数据、空目录、半迁移文件、死兼容、旧残留或会误导后续维护者的示例。发现旧设计比新真相弱，就干净删除旧设计，而不是兼容它。

完成一项任务必须做到：

- 当前产品真相清楚。
- 用户体验闭环。
- 考官和考生边界清楚。
- 题库、验证、评分、报告各自职责清楚。
- 文档、代码、测试一致。
- 可自动验证的地方已经自动验证。
- 没有影响交付的尾巴。

## 2. Owner 沟通规则

默认用用户视角回答，不用工程师视角回答。

优先说明：

- 用户会看到什么。
- Agent 会怎么参加考试。
- 考试怎么保证公平。
- 成绩怎么生成。
- Owner 怎么判断它做对了。
- 当前风险和需要决策的点。

避免说明：

- 某个函数怎么写。
- 某个文件逐行做什么。
- 某个 schema 的全部字段。
- 一大段命令输出。
- “我接下来要做什么”的长篇计划。

除非 Owner 明确要求，否则不要把实现细节当成交付说明。

## 3. 文档驱动和测试驱动

高考按文档驱动和测试驱动推进。

收到正式任务后，先理解当前 spec、项目结构、真实运行方式和已确认方向。行为变化时，先把验收点变成测试或等价验证，再实现。收口前必须让 spec、实现、测试、报告一致。

不要靠猜、靠感觉、靠“差不多”。要靠文档、测试、实现、验证和最终一致性。

## 4. 考官和考生边界

平台是考官，候选 Agent 是考生。

考生只能看到公开考卷、公开材料、公开 API 和考试协议。考生不能看到隐藏答案、验证规则、评分实现、私有 fixture、历史运行真相或其他考生痕迹。

平台拥有：

- 公开考卷。
- 机器端验证。
- 评分规则。
- 隔离环境。
- 证据记录。
- 成绩报告。

考生拥有：

- 理解题目。
- 选择行动。
- 调用自己的工具。
- 提交答案或产物。

平台不能变成考生的大脑。平台只提供考试环境、记录证据和判分。

## 5. 题库设计纪律

题库必须是真考试资产，不是假包装。

高考公开题目放在 `gaokao/tasks/`。机器端验证放在 `gaokao/validation/`。评分逻辑放在 `gaokao/scoring/`。报告生成放在报告模块。不要把答案写进公开题面，不要把题号答案硬编码进评分代码。

题库目录按验证机制组织，不按能力标签组织。执行、检索、推理、沟通等能力是题目 metadata，用于成绩单聚合。目录形态是 `gaokao/tasks/<验证类型>/<编号>/task.json`。选择题使用共享答案键；非选择题使用 `gaokao/validation/<验证类型>/<编号>/validator.ts`，一题一个验证器。

MBTI 和高考评分相互独立。MBTI 资产放在 `mbti/`，报告输出到 `.mbti/`，不得影响高考分数。

新增题型或新增题库时，优先让它成为数据资产和正式契约，而不是散落在 prompt 或临时代码里。

选择题可以作为基础题型，但不要把“好判分”误认为“足够测能力”。如果要测真实执行，就用状态检查、文件产物检查、API 状态检查、浏览器状态检查或精确答案检查。

## 6. 架构纪律

一个文件只做一件事，但不要为了拆而拆。

好结构的标准不是文件多，而是职责清楚、低耦合、高内聚、改动成本稳定。能用一句人话说清一个模块负责什么、不负责什么，才算边界清楚。

核心边界：

- `gaokao/tasks/`：能力考试公开考卷和公开材料。
- `gaokao/validation/`：机器端真相和验证数据。
- `gaokao/scoring/`：能力考试通用判分策略。
- `gaokao/src/`：能力考试服务、考试流程、状态、报告和接入表面。
- `mbti/`：独立 Agent MBTI 测试资产、服务和报告。
- `tests/`：保护产品契约和架构边界。

不要留下空目录、旧路径、旧格式、旧概念或会教错后续维护者的残留。

## 7. 真实执行和客观评分

分数只来自平台观察到的证据，不来自 Agent 自述。

优先使用确定性评分：

- 选择题标准答案。
- 精确答案。
- 文件产物检查。
- API 状态检查。
- 数据状态检查。
- 浏览器状态检查。
- 运行轨迹和证据记录。

能机器判分就不要用 LLM judge。当必须使用 judge-based 评分时，必须明确标注为较弱证据。

## 8. 本地考试体验

高考是本地考试。

服务可以调用真实 API，但交付体验必须是本地可运行、本地可验证、本地生成报告。本地成绩单默认落在：

```text
.gaokao/reports/<examId>/report.html
.gaokao/reports/<examId>/report.json
.gaokao/reports/<examId>/report.md
```

考试完成后，Agent 必须拿到报告文件，并关闭本地考务服务。不要让服务长期挂在后台。

## 9. 快速演进规则

高考默认不追求向后兼容。

当新结构更强时，旧结构必须被替换。不要保留旧任务格式、旧评分分支、旧 adapter alias、旧文档、旧测试、旧 prompt 或兼容 shim，除非 Owner 明确要求临时过渡。

标准路径是：

`新真相 -> 更新 spec -> 更新测试 -> 替换实现 -> 删除旧残留 -> 验证真实考试路径`

## 10. 验证和收口

写完不等于完成。

完成必须有真实证据：

- 相关文件已更新。
- 旧残留已删除。
- 关键测试已更新。
- 构建和测试已运行。
- 行为变化时跑过真实本地考试路径。
- 报告文件真实生成。
- 服务能正确关闭。

最终回复要短：改了什么、验证了什么、还剩什么风险。不要用大量代码细节替代产品结果。

## 11. 最终规则

永远把高考推向当前最强考试架构。

如果某个东西削弱公平性、泄露答案、制造假测试、污染用户体验、偏向某个 Agent、隐藏证据、保留死亡兼容或让后续维护更困难，删除它。

如果某个东西能提升真实考试、公平隔离、客观评分、证据质量、本地体验、用户理解和长期可维护性，干净构建并验证它。
